Three ways to let one case study serve recruiters, peers, and clients at once. Switch between them to see how each pattern shapes the reading experience. Sample content uses the Organic Flow rebuild.

Case study

Organic Flow — rebuilt for speed, control, and a CMS that doesn’t fight you

A legacy bespoke-PHP site rebuilt from the ground up in Astro on Cloudflare Pages. Faster, cheaper, easier to keep current.

The problem

The old site was slow, hard to update, and stuck in 2015 conventions.

Time-to-interactive sat around 8 seconds on simulated 3G. Lighthouse performance hovered at 42. The bespoke PHP CMS had no preview, no rollback, no per-PR staging. Any content change required a developer in the loop — so by month two of launch the site was already stale. SEO suffered from server-rendered template bloat that shipped 1.4MB of JS most readers never touched.

The rebuild

Astro on Cloudflare Pages, Python + OpenAPI backend, Apache ECharts for visuals.

Astro’s partial hydration model means the default ship is zero JS — islands opt in only where they’re needed. Content lives in structured collections, so editing copy is a Markdown edit and a CI run, not a deploy. The Python backend exposes OpenAPI, making the data contract self-documenting and typed end-to-end. Visuals render through Apache ECharts, declarative enough to keep chart logic out of components. Cloudflare Pages handles static-asset distribution at the edge and gives every PR its own preview URL.

The results

3× faster page loads, Lighthouse 100, 78% smaller JS bundle, content edits in 30 seconds.

Time-to-interactive dropped from 8.1s to 2.3s on simulated 3G. Lighthouse performance moved from 42 to 100. The JS bundle shrank by 78%. What used to be a deploy-gated content edit is now a Markdown edit that lands at the edge in under 30 seconds. Per-PR previews caught two regressions before merge in the first week alone.
