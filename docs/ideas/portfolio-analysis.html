Hero section — detailed teardown
✕
No visual anchor
A hero is the first frame a visitor sees. Without a photo, illustration, or strong typographic statement, there is nothing to form a memory around. "Tanel" in large type alone doesn't create identity.
✕
Tagline is generic and grammatically broken
"Software engineer, with data driven results." — the comma before "with" is a comma-splice error. Beyond that, "data driven results" is vague to the point of meaninglessness. Any engineer could write this. It signals nothing about domain, style, or approach.
✕
No call-to-action
A hero with no CTA is a page with no declared intent. The visitor has nowhere obvious to go. Whether the CTA is "See my work", "Get in touch", or "Download CV" depends on the audience — but something must be there.
Recommended section order
1 Hero — name, photo, specific tagline, primary CTA
2 About — 3–4 sentences, human voice, location
3 Projects — 4–6 with screenshots, stack, and outcomes
4 Skills — grouped by domain, anchored to visible project work
5 Contact — clear CTA copy, email or form, social links

No human signal P0
No photo, no avatar, no bio. Research consistently shows that portfolios without a human presence score lower on trust metrics — visitors cannot form a mental model of who they'd be working with. A face anchors everything else.
✕
Projects are underpowered P1
Two projects shown, each with a one-line description and a GitHub link. No screenshots. No stack context. No problem/outcome framing. The most important section of any developer portfolio — the proof of work — is the site's weakest section. A portfolio with zero project visuals is asking visitors to take everything on faith.

No interaction design P1
No hover states on any interactive element. No scroll-triggered animations. No micro-interactions. For a developer actively marketing infrastructure and frontend skills, a portfolio with zero interaction design is a contradiction — it implies either a lack of care or a lack of capability.
!
No defined audience or primary intent P1
The site doesn't signal who it's for — a hiring manager, a startup founder, an open-source collaborator, a freelance client. These audiences have different decision criteria and need different emphasis. Without this decision made explicitly, nothing else can be properly optimized.
!
IA order is workable but suboptimal P2
Current: Hero → Projects → Skills → Contact. Recommended: Hero → About → Projects → Skills → Contact. Skills listed after projects are contextually anchored — a visitor who's just seen what you've built will find the skills section meaningful rather than abstract.

Recommendations
Prioritized action list
P0 — before designer Fix the skills and stack content
Replace Python, Docker, BASH with TypeScript, Astro, Cloudflare Workers, Supabase, D1, GitHub Actions. Swift stays. Add categorical grouping: Frontend (Astro, TypeScript, Tailwind) · Backend (Supabase, D1) · Infrastructure (Cloudflare Workers, GitHub Actions) · Mobile (Swift, SwiftUI, AVFoundation). A designer cannot make informed layout decisions for this section without accurate content.
P0 — before designer Define the primary audience and CTA
Answer these before the first design meeting: (1) Who lands here — hiring manager, startup founder, freelance client, open-source collaborator? (2) What action do you want them to take? (3) What do you want them to feel? These three answers determine tone, hierarchy, and every layout decision that follows.
P1 — visual identity Define a color and type system
Pick one accent color that reflects personality — not blue (default) or purple (cliché for tech). Consider something warmer (ochre, rust, deep green) that stands out while feeling deliberate. Pair it with a neutral off-white or warm gray background. For typography: one display typeface with character for headings, one clean mono or sans for body. Two fonts maximum. This single decision transforms the site's perceived quality more than any layout change.
P1 — hero Add photo and rewrite the tagline
A professional photo (or carefully designed illustrated avatar). Rewrite the tagline to be specific: instead of "software engineer with data driven results," try something like "I build fast, reliable systems — Cloudflare edges to iOS apps" or "Full-stack engineer specializing in Astro, Cloudflare Workers, and Swift." Specific beats generic. Accurate beats aspirational.
P1 — projects Expand each project into a mini case study
For each of 4–6 projects: one screenshot or short demo GIF, tech stack badges (Astro / Cloudflare / Supabase / etc.), a two-paragraph problem + solution/outcome block, links to both GitHub and a live demo where available. The yolo-dualdev and Strato_Pi projects are good starting points but the current work (organicflow, crockpot recipe site, iOS video editor) is far more representative of current capability.
P2 — new section Add a short About / bio section
3–4 sentences. Not a CV summary — a human perspective. Where you're based (Warsaw), what you care about in your work, what makes your approach distinctive. Optional: a brief mention of non-work interests (dance, philosophy) creates memorability. This is what clients and collaborators actually read and remember.
P2 — interaction Add intentional hover and scroll states
Project cards should have clear hover states (subtle lift, border highlight, or color shift). Section entrances can fade-in on scroll (Intersection Observer, simple CSS). Navigation links should show active state as the user scrolls. Contact link should have a distinct hover treatment. These are table stakes for a developer portfolio in 2026 and should cost the designer nothing to spec.
P3 — SEO enhancement Add JSON-LD Person structured data
A small script block in the head helps search engines understand the page represents a specific person. Include: name, url, jobTitle, sameAs (LinkedIn, GitHub). Takes 10 minutes to add to the Astro layout and provides meaningful semantic signal to crawlers. The existing OG/meta work is good — this completes it.
